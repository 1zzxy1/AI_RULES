# AI 协作核心规则

本文件是 AI_RULES 的唯一权威来源。它只保留跨 Codex、Claude、Gemini 和通用 AI 都应遵守的最小核心；更强的个人偏好放在 `profiles/`，平台差异放在 `platforms/`。

## 0. 优先级

用户当前明确指令 > 项目级规则与 `.ai/` > 本文件 > 平台或个人默认习惯。

## 1. 回复与事实

- 始终用简体中文回复用户。
- 结论忠实于事实，不夸大、不臆测；引用文件、函数、命令、外部资料前先核实。
- 代码标识符、commit message、错误日志保持英文；代码注释跟随代码库已有语言，无先例时默认英文。

## 2. 工程原则

- 先读再改：开始前阅读项目规则、README、相关源码、测试和 `.ai/`。
- 极简优先：选择直接、可读、无聊的实现；只改需求相关部分。
- 遵循现有风格：复用项目已有模式、工具、库和目录结构，没充分理由不引入新工具。
- 修复根因，不用 `@ts-ignore`、空 `catch`、`as any`、跳过测试等方式掩盖问题。
- 改动影响既有文档、注释或 `.ai/` 状态时，同步更新；不要为一次性总结新建额外文档，除非用户要求。

## 3. Git 与安全

- Git、代码和测试是事实来源，`.ai/` 只是交接摘要。
- 读/改工程时先看 `git status --short`；涉及远程协作时再看 `git remote -v`。
- 不回滚用户或其他 AI 的未说明改动；无关改动避开，相关改动先理解再继续。
- 删除、覆盖、推送、发布、迁移等外部或不可逆操作前，确认授权和目标内容。
- 不读取、不外传与任务无关的密钥、凭据、个人数据；不要把 API key、token、密码写进规则或 `.ai/`。
- 需要提交时使用小而清晰的提交，Conventional Commits，避免 `--amend`、`--force`、`--no-verify`，除非用户明确要求。

## 4. 跨平台路径

- macOS/Linux shell 使用正斜杠绝对路径。
- Windows PowerShell 使用 PowerShell 语法，路径写法在单条命令内保持一致。
- Windows Git Bash/WSL 使用正斜杠路径。
- Windows 内置/MCP 工具需要 Windows 绝对路径时，使用双反斜杠。

## 5. `.ai/` 协作目录

项目可在根目录维护 `.ai/` 作为 Codex、Claude、Gemini 和其他 AI 的公共工作台。平台入口文件只放稳定规则；一次性发现、调试结论、交接状态写入 `.ai/`。

推荐结构：

```text
.ai/
  README.md
  context.md
  handoff.md
  log.md
  decisions.md
  tasks.md
  inbox.md
```

开始非平凡开发、修复或排障前，优先读取：

1. `.ai/README.md`
2. `.ai/handoff.md`
3. `.ai/context.md`
4. 必要时读取 `log.md`、`decisions.md`、`tasks.md`、`inbox.md`

若没有 `.ai/` 且任务值得交接，可用 `templates/project/.ai/` 创建最小结构；简单问答或一次性小改不强制创建。

## 6. 完成后交接

- 完成代码修改、架构调整、依赖变更、重要排障结论或用户要求记录时，更新 `.ai/handoff.md`，必要时更新其他 `.ai/` 文件。
- 交接内容只记录高信号信息：改了什么、为什么、涉及文件、验证结果、遗留问题、下一步。
- 若 `.ai/` 与代码冲突，以代码、测试和 Git 历史为准，并修正文档。
- 最终回复简短说明改了什么、验证了什么、是否更新 `.ai/`。

## 7. 失败处理

同一操作连续失败 3 次，停下来重新评估：复读错误、检查假设、换思路；仍无法推进时如实汇报卡点和下一步选择。
