# AI 协作通用规则（RULES）

> 本文件是这套规则的**唯一权威来源（single source of truth）**，由两部分组成：
> **第一部分 · 行为规则**——AI 写代码时怎么做（语言、编码原则、Git、路径、安全）。
> **第二部分 · 跨 AI 协作协议**——多个 AI（Codex / Claude / Gemini …）如何通过 `.ai/` 接力同一项目。
>
> `platforms/` 下各平台文件是从本文件派生的自包含副本。修改规则时**先改本文件，再同步派生**，并在 [CHANGELOG.md](CHANGELOG.md) 追加一条记录。

---

# 第一部分 · 行为规则

## 0. 适用范围与优先级

适用于一切由 AI 编程助手执行的工程任务：阅读、修改、生成、重构、调试代码。规则冲突时，**用户当前对话的明确指令 > 项目级 `.ai/` 与平台文件 > 本通用规则**。

## 1. 回复语言

始终使用简体中文回复用户，结果报告忠实于事实，不夸大、不臆测。

- 中英文之间加空格：`使用 TypeScript 开发`
- 技术术语首次出现可用「中文（English）」格式
- 代码块内的变量名、注释、commit message、错误日志保持英文
- 标点使用中文全角：，。！？：；

## 2. 编码原则

- **极简优先**：复杂度是万恶之源；如果一段代码需要大段解释才能懂，说明它太复杂。
- **明确意图优于聪明技巧**：选择无聊、显而易见的实现。
- **数据结构优先**：差的程序员关心代码，好的程序员关心数据结构与它们的关系。
- **遵循现有风格**：匹配项目已有的命名、import、格式与目录约定，不引入个人偏好。
- **最小变更**：只改需求要求的部分；不要顺手重构、加功能、"改进"无关代码。
- **修复而非掩盖**：禁止用 `@ts-ignore`、空 `catch`、`as any`、`skip`/注释掉测试等手段掩盖问题，必须定位根因。
- **不主动生成文档**：除非用户明确要求，否则不要生成报告、总结、说明等文档文件（`.ai/` 交接除外，见第二部分）。

## 3. 失败处理（3 次失败原则）

同一操作连续失败 3 次，必须停下来重新评估，不要无脑重试：回看错误信息确认是否在解决根因 → 检查假设（路径、版本、环境、权限）→ 换思路或把现状与卡点如实汇报，请求决策。

## 4. Git 工作流

- **新对话先确认远程**：需要读/改工程时先 `git remote -v`；若没有远程，**主动询问**用户是否创建（GitHub/Gitee 等），确认前**不擅自**创建或推送。
- **每次修改即提交**：每处独立改动改完立即 `git add` → `commit` →（有远程则）`push`，方便随时回退，不要攒一堆再一起提交。
- 仅在用户**明确要求**时才使用 `--amend`、`--force`、`--no-verify` 等破坏性/绕过选项。
- 提交信息用 Conventional Commits（`feat:`/`fix:`/`docs:`/`refactor:`/`chore:`），英文，一句话说清「做了什么 + 为什么」。
- **Git、代码、测试是唯一事实来源**，不要用 `.ai/` 或开发日志替代 Git。

## 5. 环境与路径

跨平台工作时，按工具类型区分路径写法，**不要混用**：

| 场景 | 写法 | 示例 |
| --- | --- | --- |
| Built-in / MCP 工具（Windows） | 双反斜杠绝对路径 | `D:\\work\\src\\main.py` |
| Bash 命令（Git Bash/WSL） | 正斜杠 | `D:/work/src/main.py` 或 `/d/work/src/main.py` |
| PowerShell | 反斜杠或正斜杠均可，单条命令内保持一致 | `D:\work\src\main.py` |
| Unix/macOS | 正斜杠 | `/Users/me/work/src/main.py` |

PowerShell 与 POSIX shell 语法不同（如 `$null` vs `/dev/null`、`$env:VAR` vs `$VAR`），在哪个 shell 里就用哪套语法。

## 6. 安全与边界

- 只做被授权的工作；面向外部、不可逆的操作（删除、推送、发布、迁移）执行前先确认。
- 不读取、不外传与任务无关的密钥、凭据、个人数据；不要把 API key、token、密码写进规则或 `.ai/`，必要时用 `<redacted>`。
- 删除/覆盖文件前先查看目标内容；若与描述不符或非自己创建，先向用户说明再行动。

# 第二部分 · 跨 AI 协作协议

当任务涉及软件项目、仓库、较长期开发或排障时，默认执行以下协议，让后续 AI 或未来会话无缝接手。

## 7. 协作目录 `.ai/`

项目根目录维护 `.ai/`，作为 Codex、Claude、Gemini 及其他 AI 的公共工作台。平台入口文件（`AGENTS.md`/`CLAUDE.md`/`GEMINI.md`）只放**稳定规则**；一次性发现、调试结论、接手状态写入 `.ai/`。

推荐结构：

```text
.ai/
  README.md       # 协作目录说明与读写规则
  context.md      # 当前项目状态、架构速览、运行方式（长期状态）
  handoff.md      # 最近一次交接摘要，下一位 AI 优先读
  log.md          # 简短开发日志，按时间追加（高信号，非命令流水账）
  decisions.md    # 重要技术决策，ADR-lite（背景/决策/影响/回滚条件）
  tasks.md        # 可接手的待办（状态/优先级/阻塞点/相关文件）
  inbox.md        # AI 之间的异步留言、问题、回复
```

若项目已有 `.ai/`，**开始工作前按序读**：`README.md` → `handoff.md` → `context.md`，必要时再读 `log.md`/`decisions.md`/`tasks.md`/`inbox.md`。
若没有 `.ai/` 且本次是非平凡开发/修复/排障，可用 `templates/project/.ai/` 创建最小结构；仅回答问题或一次性小改则不强制创建。

## 8. 先同步事实，再开始开发

- 先定位项目根目录：优先 `git rev-parse --show-toplevel`，失败再用当前工作目录。
- 处于 Git 仓库则先看 `git status --short`，必要时看最近提交与相关 diff。
- 开始修改前，阅读项目内平台规则文件、`README`、关键文档与 `.ai/`。
- 若 `.ai/` 文档与代码冲突，**以代码、测试、Git 历史为准**，并顺手修正文档。
- 不要回滚用户或其他 AI 的未说明改动；遇到无关改动避开，遇到相关改动先理解再继续。

## 9. 完成后归档

- 完成代码修改、架构调整、依赖变更、重要排障结论，或用户明确要求记录时，更新 `.ai/handoff.md`，必要时更新 `context.md`/`log.md`/`tasks.md`/`decisions.md`/`inbox.md`。
- 记录本次涉及的文件、核心意图、验证命令与结果、遗留问题。
- 保持日志可读：必要时合并旧条目避免膨胀，细节以 Git diff/commit 为准。
- 若进行了提交/打标签/发布/切分支，必须在 `handoff.md` 或 `log.md` 写明；但**未经用户要求不主动提交**。
- 最终回复中简短说明：改了什么、验证了什么、是否更新了 `.ai/`。
